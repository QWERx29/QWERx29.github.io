哈希表

本文最后更新于：2022年8月11日 中午

哈希表（Hash Table）又称散列表，是一种以 key-value 形式存储数据的数据结构。

所谓以 key-value 形式存储数据，是指任意的 key 都唯一对应到内存中的某个位置。只需要给出查找的值 key，就可以快速地找到其对应的 value。

可以把哈希表理解为一种高级的数组，这种数组的下标可以是大整数，浮点数，字符串，甚至结构体。

哈希函数

要让 key 对应到内存中的位置，就要为 key 计算索引，也就是计算 value 的存储位置。根据 key 计算索引的函数就叫做哈希函数，也称散列函数。

在 OI 中，最常见的情况是 key 为整数的情况。当 key 的范围比较小的时候，可以直接把 key 作为数组的下标；但当 key 的范围比较大，就需要用到哈希表。一般把 key 模一个较大的质数作为索引，也就是取 f(x)=xmod⁡Mf(x)=x\operatorname{mod} Mf(x)=xmodM 作为哈希函数。

另一种可能出现的情况是 key 为字符串的情况，但在 OI 中，一般不直接把字符串作为 key，而是先算出字符串的哈希值（将在下一节中介绍），再把其哈希值作为 key 插入到哈希表里。

能为 key 计算索引之后，我们就可以知道每个 value 应该放在哪里了。假设我们用数组 aaa 存放数据，哈希函数是 f\mathcal{f}f ，那键值对 (key,value)(key,value)(key,value) 就应该放在 a[f(key)]a[\mathcal{f}(key)]a[f(key)] 上。

无论 keykeykey 是什么类型，范围多大，f(key)\mathcal{f}(key)f(key) 都应是在可接受范围内的整数，可以作为数组的下标。

冲突

对于任意的 keykeykey ，如果哈希函数计算出来的索引都不相同，那只用根据索引把 (key,value)(key,value)(key,value) 放到对应的位置就行了。但实际上，常常会出现两个不同的 keykeykey ，他们用哈希函数计算出来的索引是相同的。这时候就需要一些方法来处理冲突，常用的方法是拉链法。

拉链法

拉链法也称开散列法（open hashing）。

拉链法是在每个存放数据的地方建立一个链表，如果有多个 keykeykey 索引到同一个地方，只用把他们都放到那个位置的链表里就行了。查询的时候需要把对应位置的链表整个扫描一遍，对其中的每个数据比较其 keykeykey 与查询的 keykeykey 是否一致。如果索引的范围是 0∼M−10 \sim M-10∼M−1 ，哈希表的大小为 NNN ，那么一次查询需要进行期望 O(NM)\mathcal{O}(\dfrac{N}{M})O(MN​) 次比较。

闭散列法

闭散列方法把所有记录直接存储在散列表中，如果发生冲突则根据某种方式继续进行探查。

比如线性探查法：如果在 d 处发生冲突，就依次检查 d + 1 ， d + 2 ……

当散列表接近存满时，查找效率较差；存满后不能再插入新元素。

比赛中直接使用 <unordered_map> 即可！

字符串 Hash

字符串 Hash 是一种高效处理字符串匹配问题的近似算法，其核心思想在于将难以快速比较的字符串映射到一个值域较小（但也不能过小）、可以方便比较的范围。

我们定义一个把字符串映射到整数的函数 f\mathcal{f}f ，这个 f\mathcal{f}f 称为是字符串 Hash 函数。

我们希望这个函数 f\mathcal{f}f 可以方便地帮我们判断两个字符串是否相等。具体来说，我们希望在 Hash 函数值不一样的时候，两个字符串一定不一样。需要注意的是：其否命题不一定总是成立，也就是说：不相同的字符串经过 Hash 函数映射也可能得到相同的值，这种现象称作 Hash 冲突。

通常我们采用的是多项式 Hash 的方法，即 f=∑s[i]×basei(mod⁡MOD)\mathcal{f}=\sum s[i]\times base^i (\operatorname{mod} MOD)f=∑s[i]×basei(modMOD) 或 f(s)=∑s[i]×base∣s∣−i(modMOD)f(s) = \sum s[i] \times {base}^{|s|-i} \pmod{MOD}f(s)=∑s[i]×base∣s∣−i(modMOD) （虽然前一种看起来更简洁，但在实际使用中会出现麻烦的“对齐问题”，因此更常用实际上的是后一种定义）。

这里面的 basebasebase 和 MODMODMOD 需要选取得足够合适才行，以使得 Hash 函数的值分布尽量均匀，尽量避免发生 Hash 冲突。

如果 basebasebase 和 MODMODMOD 互质，在输入随机的情况下，这个 Hash 函数在 [0,M)[0,M)[0,M) 上每个值概率相等，此时单次比较的冲突率为 1M\dfrac{1}{M}M1​ 。所以，哈希的模数一般会选用大质数。

冲突率

若进行 nnn 次比较，每次冲突率 1M\dfrac{1}{M}M1​ ，那么总冲突率是 1−(1−1M)n1-\left(1-\dfrac 1 M\right)^n1−(1−M1​)n 。在随机数据下，若 M=109+7,n=106M=10^9 + 7,n=10^6M=109+7,n=106 ，冲突率约为 11000\dfrac 1{1000}10001​ ，并不是能够完全忽略不计的。

一种特殊的情形是利用 unsigned long long 类型的自然溢出特性（相当于对 M=264M = 2^{64}M=264 取模），这能有效简化代码实现。但有经验的出题人很可能会对此定向构造数据，显著提高自然溢出的冲突率，甚至确保发生 Hash 冲突。

因此，进行字符串哈希时，为了安全起见，经常会分别取两组不同的 basebasebase 和 MODMODMOD 分别计算 Hash 值，这样哈希函数的值域就能扩大到两者之积，错误率就能进一步减小了。但由于这种写法较为繁琐，所以一般情况下还是会只使用一组质数或者干脆使用自然溢出。

多次询问子串哈希

单次计算一个字符串的哈希值复杂度是 O(n)\mathcal{O}(n)O(n) ，其中 nnn 为串长，与暴力匹配没有区别，如果需要多次询问一个字符串的子串的哈希值，每次重新计算效率非常低下。

一般采取的方法是对整个字符串先预处理出每个前缀的哈希值，将哈希值看成一个 bbb 进制的数对 MMM 取模的结果，这样的话每次就能快速求出子串的哈希了：

令 fi(s)f_i(s)fi​(s) 表示 f(s[1..i])f(s[1..i])f(s[1..i]) ，那么 f(s[l..r])=fr(s)−fl−1(s)×br−l+1f(s[l..r])=f_r(s)-f_{l-1}(s) \times b^{r-l+1}f(s[l..r])=fr​(s)−fl−1​(s)×br−l+1 ，其中 br−l+1b^{r-l+1}br−l+1 可以预处理出来。

这样就可以在 O(n)\mathcal{O}(n)O(n) 的预处理后每次 O(1)\mathcal{O}(1)O(1) 地计算子串的哈希值了。

字符串 Hash 的应用

字符串匹配

求出模式串的哈希值后，求出文本串每个长度为模式串长度的子串的哈希值，分别与模式串的哈希值比较即可。

字符串最小循环节

引理

字符串 s[1⋯n]s[1\cdots n]s[1⋯n] 的循环节为 ddd 的充分必要条件为 s[1..n−d]=s[d+1..n]s[1..n-d]=s[d+1..n]s[1..n−d]=s[d+1..n] 。

读者自证不难。

枚举字符串长度的所有约数，使用 Hash 判断即可。

特殊变换

最长回文子串

二分答案，判断是否可行时枚举回文中心（对称轴），哈希判断两侧是否相等。需要分别预处理正着和倒着的哈希值。时间复杂度 O(nlog⁡n)\mathcal{O}(n\log n)O(nlogn) 。
